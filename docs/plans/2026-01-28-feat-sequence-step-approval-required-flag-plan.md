## Overview

Add a per-step `approval_required` flag to sequence steps that controls whether AI-personalized emails require human approval before sending. When approval is not required, emails are auto-approved immediately after generation.

## Problem Statement / Motivation

Currently, all emails generated by the sequence scheduler go to the Outbox with `status='pending'` and require manual human approval before sending. This creates a bottleneck for organizations that trust their AI personalization and want to automate the full flow.

Users need the ability to configure which steps require human review and which can be auto-approved, giving them control over the trade-off between oversight and automation speed.

## Proposed Solution

1. Add `approval_required` boolean column to `sequence_steps` table (default `true`)
2. Modify sequence scheduler to create outbox with `status='approved'` when `approval_required=false`
3. Add UI toggle in sequence builder to configure the setting per step

## Technical Approach

### Data Flow

```
[Sequence Scheduler]
       |
       v
   Check step.approval_required
       |
       +---> TRUE: Create outbox (status='pending') [CURRENT BEHAVIOR]
       |
       +---> FALSE: Create outbox (status='approved', reviewed_by=NULL)
                    Then create email_sends record (same transaction)
```

### Research Insights: Architecture

**Key Decision: Create with final status directly**

Instead of INSERT (pending) + UPDATE (approved), create the outbox item with its final status. This:
- Eliminates one database round-trip per auto-approved item
- Keeps the status machine clean (no transient states)
- Avoids race conditions

**Why column vs JSONB is correct:**
- Query pattern alignment: scheduler checks this flag for every enrollment
- Existing pattern consistency: `default_delay_hours`, `subject` are columns, not in `builder_content`
- Type safety and schema documentation

### Database Changes

**Migration: `031_sequence_step_approval_required.sql`**

```sql
-- Migration: Add approval_required to sequence_steps
-- Purpose: Allow sequence steps to bypass human approval

BEGIN;

-- Safety: Set short lock timeout to fail fast if tables are busy
SET LOCAL lock_timeout = '3s';

-- Add approval_required to sequence_steps
-- PostgreSQL 11+: No table rewrite for static defaults (instant operation)
ALTER TABLE sequence_steps
ADD COLUMN approval_required BOOLEAN NOT NULL DEFAULT true;

COMMENT ON COLUMN sequence_steps.approval_required
    IS 'When false, emails from this step bypass human approval and send automatically';

COMMIT;
```

### Research Insights: PostgreSQL Migration

**Best Practices Applied:**
- `SET LOCAL lock_timeout = '3s'` - Fails fast if table is locked by long transaction
- PostgreSQL 11+ fast path - Adding NOT NULL column with static default doesn't require table rewrite
- Lock is held only briefly (milliseconds) for metadata change
- `COMMENT ON COLUMN` - Documents the bypass behavior

**Why no partial index needed:**
- A partial index on `WHERE auto_approved = true` indexing only the boolean itself is wasteful
- The indexed value would always be `true` - no filtering power
- If we ever need to query auto-approved items, use: `WHERE status = 'approved' AND reviewed_by IS NULL`

**File: `functions/sequence_scheduler/handler.py`**

Update outbox creation logic (~line 180-189):
```python
# Determine if auto-approval is enabled
approval_required = step_content.get("approval_required", True)
outbox_status = "pending" if approval_required else "approved"

# Create outbox item with final status
outbox_item = await OutboxQueries.create(
    conn,
    org_id=org_id,
    target_id=enrollment["target_id"],
    channel="email",
    subject=subject,
    body=body,
    priority=5,
    status=outbox_status,
)

# If auto-approved, create email_sends record immediately
if not approval_required:
    await conn.execute(
        """
        INSERT INTO email_sends
            (organization_id, target_id, email, subject, body, status, outbox_id)
        SELECT $1, $2, t.email, $3, $4, 'queued', $5
        FROM targets t WHERE t.id = $2
        """,
        org_id,
        enrollment["target_id"],
        subject or "",
        body,
        outbox_item["id"],
    )
    logger.info(f"Auto-approved outbox item {outbox_item['id']} for enrollment {enrollment['id']}")
```

### Research Insights: Transaction Safety

**Atomic operation pattern:**
- Both outbox creation and email_sends insert happen within the existing transaction context
- If email_sends insert fails, the entire transaction rolls back (including outbox)
- No partial state possible

**Avoid these pitfalls:**
- Don't use connection after transaction context exits
- Don't start async tasks that outlive the transaction
- Keep slow operations (like Maven API calls) outside the transaction

### Research Insights: UX Best Practices

**Why inline warning instead of confirmation dialog:**
- NN/g research: toggles expect immediate results; dialogs break mental model
- Confirmation dialogs cause "dialog fatigue" and get ignored
- This is a reversible preference setting, not a destructive action

**Accessibility requirements (W3C WAI-ARIA):**
- Always associate Label with Switch via `htmlFor`
- Keep label text static ("Require approval"), change description based on state
- Radix Switch handles keyboard navigation automatically

## Security Considerations

### Acknowledged Risks (User's Explicit Choice: No Safeguards)

The user explicitly chose "No safeguards - auto-approve immediately." The following risks are accepted:

1. **AI content sent without review** - Malicious or incorrect AI output could be sent
2. **No rate limiting** - Runaway automation could send many emails quickly
3. **No org-level override** - Any user with step edit permissions can enable auto-approve

### Future Safeguard Options (If Needed Later)

If issues arise, consider adding:
- **Content moderation** - AWS Comprehend or similar before auto-approval
- **Org-level daily limits** - `daily_email_limit` column on organizations
- **Admin override** - `require_human_approval` flag on organizations that overrides step setting
- **Anomaly detection** - Alert if send rate exceeds 2x historical baseline
